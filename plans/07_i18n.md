# Plan 07 — Internationalization (i18n)

## Supported Locales
| Code | Language | Trigger                          |
|------|----------|----------------------------------|
| `en` | English  | Default / fallback               |
| `ko` | Korean   | `navigator.language` starts with `ko` |

---

## Two-Layer i18n Architecture

This project has **two separate translation concerns**:

| Layer              | What it covers                              | Location              |
|--------------------|---------------------------------------------|-----------------------|
| **UI Strings**     | Labels, placeholders, tooltips, errors      | `src/locales/`        |
| **Command Data**   | `title`, `description`, `aliases` per cmd  | `src/data/ko/` + `en/`|

These must stay **completely separate**. UI strings use `i18next`. Command data uses the merge strategy from `Plan 03`.

## i18next Configuration (`src/main.tsx`)

```ts
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import enTranslation from './locales/en/translation.json';
import koTranslation from './locales/ko/translation.json';
import { detectUserLocale } from './utils/locale';

i18n.use(initReactI18next).init({
  resources: {
    en: { translation: enTranslation },
    ko: { translation: koTranslation },
  },
  lng: detectUserLocale(),   // Auto-detect on first load
  fallbackLng: 'en',
  interpolation: {
    escapeValue: false,      // React already escapes
  },
});
```

## Locale Detection (`src/utils/locale.ts`)

```ts
const LOCALE_STORAGE_KEY = 'oms-locale';

export function detectUserLocale(): 'en' | 'ko' {
  // 1. Check user's saved preference first
  const saved = localStorage.getItem(LOCALE_STORAGE_KEY);
  if (saved === 'en' || saved === 'ko') return saved;

  // 2. Auto-detect from browser/OS
  const browserLocale = navigator.language ?? navigator.languages?.[0] ?? 'en';
  const isKorean = browserLocale.toLowerCase().startsWith('ko');

  return isKorean ? 'ko' : 'en';
}

export function saveLocalePreference(locale: 'en' | 'ko'): void {
  localStorage.setItem(LOCALE_STORAGE_KEY, locale);
}
```

## Single Source of Truth: `useUIStore.language`

**Problem with dual state:** Both `i18next.language` and `useUIStore.language` would hold the locale.
Manual 3-step sync on every toggle is brittle — one missed call = UI in Korean but search data in English.

**Solution:** `useUIStore.language` is the single source of truth.
`i18next` is kept in sync via a one-way subscription, never driven directly.

```ts
// src/store/useUIStore.ts
import { create } from 'zustand';
import { subscribeWithSelector } from 'zustand/middleware';
import i18n from 'i18next';
import { saveLocalePreference } from '@/utils/locale';

// store is pure state — no side effects inside
// subscribeWithSelector middleware enables selector-based subscriptions (required for the subscriber below)
const useUIStore = create<UIState>()(
  subscribeWithSelector((set) => ({
    language: 'en',
    setLanguage: (lang: 'en' | 'ko') => set({ language: lang }),
  }))
);

// Side effects declared ONCE outside the store, using the selector-based subscribe
// This keeps the store unit-testable (no i18n or localStorage mocking needed)
useUIStore.subscribe(
  (state) => state.language,          // selector: only react to language changes
  (lang) => {
    i18n.changeLanguage(lang);        // sync i18next (UI strings)
    saveLocalePreference(lang);       // persist to localStorage
    invalidateFuseCache();            // clear stale Fuse indexes for old language
  },
  { equalityFn: Object.is },
);
// Note: requires Zustand's subscribeWithSelector middleware on useUIStore
```

```ts
// src/main.tsx — init sequence (order matters)
const detectedLocale = detectUserLocale();
await i18n.init({ lng: detectedLocale, ... });         // 1. init i18n first
useUIStore.getState().setLanguage(detectedLocale);      // 2. triggers subscriber → idempotent
```

## Language Toggle Behavior

```
User opens site
  → detectUserLocale() → 'ko'
  → useUIStore.setLanguage('ko')
      ├─ sets store.language = 'ko'
      ├─ calls i18n.changeLanguage('ko')  (UI strings update)
      └─ saves 'ko' to localStorage
  → useCommandSearch reads store.language → uses ALL_COMMANDS_KO

User clicks EN toggle
  → useUIStore.setLanguage('en')
      ├─ sets store.language = 'en'
      ├─ calls i18n.changeLanguage('en')
      └─ saves 'en' to localStorage
  → useCommandSearch re-derives from ALL_COMMANDS_EN (Fuse re-indexes)
  → ResultList re-renders with English descriptions
```

## Command Data Language Sync

`useCommandSearch` reads language from the store — automatically re-runs when language changes:

```ts
// In useCommandSearch.ts
const language = useUIStore(s => s.language);
const allCommands = language === 'ko' ? ALL_COMMANDS_KO : ALL_COMMANDS_EN;
// ↑ useMemo dependency on allCommands → Fuse index rebuilds on language switch
```

## Rules
- `t('key')` only for UI strings — never for command content
- Never hardcode any user-visible string in components — always use `t()`
- `translation.json` keys must be identical between `en/` and `ko/`
- If a `ko/` command entry is missing, the merger silently falls back to `en/` — no error
